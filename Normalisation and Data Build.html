Normalisation and Data Build

Using sample student data to demonstrate normalisation and database design skills.

Raw Data

The Student data is not normalised. As we can see from the table on the left, each student record has multiple courses in the same row. In order to convert this to the first normal form we need to ensure that each cell is Atomic, meaning it holds a single value, splitting the courses into distinct rows for each student.

First Normal Form

The improved table on the right demonstrates First Normal Form, meeting the conditions for Atomicity. Each Course, Exam Board and Teacher now appear seperate records, rather than under the same one. However, we can see that the Course, Exam Board and Teacher are not dependent on the Student ID, which is the primary key.

Second Normal Form

In order to conform to Second Normal Form we need to remove these partial functional dependencies. For this, I have split the Course, Exam Board and Teacher fields into a separate table, using a new ID field as a primary key and the Student Number field as a Foreign Key. However we still have some transitive functional dependencies. For example, the Exam Board and Teacher fields are still determined by the course and not the ID.

Third Normal Form

The Third Normal Form is met by splitting off the Exam Board Field into it's own table and splitting Course Name and Teacher Name into their own table. We generate a new Primary Key for each of these and link them to a Foreign Key in the Courses table. This eliminates any transitive functional dependencies and minimises data duplication.
